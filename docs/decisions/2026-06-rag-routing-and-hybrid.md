# RAG 路由与 Hybrid Retrieval 决策

## 背景

- “一共有多少篇周报？”属于聚合统计，不应依赖分块相似度检索。
- Vercel 原先禁用查询向量，线上实际只有关键词检索。
- 空召回直接返回统一提示，缺少查询改写和标题元数据兜底。

## 决策

- 先做任务路由：周报总数、月份数量、最近 N 份和最近一周内容走确定性路径。
- 内容问答使用 BM25 + 同模型向量候选合并，并按章节质量做轻量重排。
- 查询向量与索引模型必须完全一致；Vercel 不加载本地 Transformer。
- 线上索引使用 `gemini-embedding-2`，通过现有 Gemini 网关生成 384 维向量。
- 主检索为空时只改写一次常见中英文术语，再检索文档标题和目录元数据。
- 英文专名必须在证据中真实出现，避免 GraphRAG 等未收录主题被语义近邻误召回。

## 未采用方案

- GraphRAG：当前问题主要是统计路由与基础召回，不需要实体关系图。
- Agentic RAG：当前问答无需多工具、多步骤计划，复杂度和延迟不划算。
- 混用本地与远程向量：不同向量空间不可直接比较，会产生不可解释的排序。

## 风险与缓解

- Gemini embedding 有每分钟配额：`npm run reembed` 按批限速、429 重试并记录断点。
- 链接表和统计表可能挤占主题证据：内容问题对低信息章节降权。
- 远程向量不可用：自动降级为关键词检索，界面按实际能力显示检索模式。

## 回滚

- 回滚本次代码提交即可恢复旧路由与检索逻辑。
- 索引是部署快照；需要回滚向量时，恢复旧 `data/index.json` 并重新部署，无数据迁移。
